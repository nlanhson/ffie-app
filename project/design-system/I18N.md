# FFIE Internationalization

**Version:** 1.0
**Last updated:** 2026-05-28
**Status:** Foundation wired — French is the default URL, English at `/en/*`. Catalog seeded with Nav + Overview strings; remaining pages migrate string-by-string.

---

## 1. Locale policy

**Supported locales:** `fr`, `en`.
**Default:** `fr` (French — FFIE is a French federation; Julien and Sylvie both work in French).
**URL strategy:** `localePrefix: "as-needed"`.
- French → `/`, `/foundations/color`, `/components/input` (no prefix, French is the canonical URL).
- English → `/en`, `/en/foundations/color`, `/en/components/input` (prefixed).
- Requesting `/fr/*` 307-redirects to the prefix-stripped URL and sets `NEXT_LOCALE=fr`.

The defaultLocale-without-prefix policy makes the URL read as a French-federation tool by default; English is opt-in.

## 2. Library + setup

- **`next-intl` v4.12** — the canonical i18n library for the Next.js App Router. Handles message catalogs, locale-aware navigation, server-component translations, ICU MessageFormat (variables, pluralization, dates, numbers).
- **Middleware** at [`src/middleware.ts`](../../design-system-preview/src/middleware.ts) handles locale detection + URL rewriting. **Must live under `src/`** (not the project root) because this project uses a `src/` directory.
- **Routing config** at [`src/i18n/routing.ts`](../../design-system-preview/src/i18n/routing.ts).
- **Request config** at [`src/i18n/request.ts`](../../design-system-preview/src/i18n/request.ts) — loads the message catalog for the current locale, sets `timeZone: "Europe/Paris"`.
- **Locale-aware navigation helpers** at [`src/i18n/navigation.ts`](../../design-system-preview/src/i18n/navigation.ts) — `Link`, `useRouter`, `usePathname`, `redirect` that automatically prefix the active locale when needed.
- **next.config.mjs** wraps the Next config with `createNextIntlPlugin("./src/i18n/request.ts")`.

## 4. Translation conventions

### Naming
Catalog keys use **PascalCase namespace + camelCase keys**, mirroring the React component tree:

```json
{
  "Nav": { "overview": "Vue d'ensemble", "color": "Couleurs" },
  "Overview": { "title": "Système de design FFIE", "tagline": "Construisons l'électricité, ensemble." }
}
```

Each component reads its namespace via `useTranslations("Nav")` and references keys without the namespace prefix: `t("overview")`.

### Variables, dates, numbers
Use ICU MessageFormat:
- Variables: `"updated": "Mis à jour le {date}"` → `t("updated", { date })`
- Dates: pass a `Date` and use the formatter: `t("updated", { date: new Date() })` — `next-intl` formats per the active locale and `Europe/Paris` timezone.
- Numbers / currencies: `t("memberCount", { count })` with `{count, number}` in the catalog.

### Pluralization
ICU plural rules:
```json
{ "docCount": "{count, plural, =0 {Aucun document} one {# document} other {# documents}}" }
```

French has different plural rules than English; ICU handles this. Always use the plural construct — never concatenate.

### Locale-aware navigation
**Always** use `Link` from `@/i18n/navigation` for internal links. The locale-aware version automatically prefixes `/en/` when on the English route and strips it on French. Plain `next/link` will break the locale boundary.

## 5. French typographic conventions

These are content-level rules. Designers should follow them when writing FR copy:

- **Non-breaking space before `:`, `;`, `?`, `!`, `«»`.** In Markdown / JSX, use ` ` or `&nbsp;`. Example: `Suivant :` not `Suivant :`.
- **Guillemets** (« » with NBSP inside): `« texte »` — preferred over straight quotes.
- **Decimal comma:** `1 234,56` — space as thousands separator, comma as decimal.
- **Date format:** `28 mai 2026` (no comma, lowercase month, no zero-padding on day).
- **Capitalization:** lowercase the day name and month name unless at sentence start.

`next-intl`'s built-in date/number formatters respect these for the `fr` locale automatically. The only thing they don't do is the NBSP-before-punctuation rule — that's manual.

## 6. Accent + character coverage

- **next/font Montserrat** is configured with `subsets: ["latin", "latin-ext"]` — covers every French accent (`à â ä é è ê ë î ï ô ö ù û ü ÿ ç œ`) plus `Ł` `š` etc. ✓
- **Museo Sans** (paid, license TBC) — if FFIE confirms a webfont license, verify the bundle includes `latin-ext`. Most paid foundries split subsets and charge per-subset.

## 7. Layout density at 200% French

Per TYPOGRAPHY.md §5 (Dynamic-Type contract): French is ~15-25% longer than English on average. **At 200% scale (WCAG SC 1.4.4) French is effectively English at ~230%.** Every layout that passes the 200% test in English must be re-validated in French. The Dynamic-Type simulator at `/foundations/typography` is locale-aware — switch to FR and re-run the 200% toggle.

## 8. Migration plan (pages still on hardcoded strings)

Today the seed catalog covers Nav + Overview only. Every other page renders its hardcoded English strings under both locales. The migration is one-page-at-a-time:

1. Identify the strings in the page (manual or grep).
2. Add a section to both `messages/fr.json` and `messages/en.json` with the keys.
3. Convert the page to a Server Component (if not already), `import { getTranslations } from "next-intl/server"` and `const t = await getTranslations("PageName")`.
4. For client components: `useTranslations("PageName")`.
5. Replace string literals with `t("key")` calls. Use ICU MessageFormat for variables / plurals.
6. Verify the page renders correctly under both `/foo` and `/en/foo`.

**Pages currently using hardcoded strings:** every page under `[locale]/` except the Overview is bilingual-by-content (component spec docs, the Julien find-doc flow with French sample text, etc.) and renders identically in both locales today. They'll be migrated as their content evolves.

## 9. The locale switcher

[`src/components/LocaleSwitcher.tsx`](../../design-system-preview/src/components/LocaleSwitcher.tsx) is a client component mounted in the Nav. It uses `useRouter().replace(pathname, { locale })` to switch the URL while preserving the current route. The browser sends an updated `NEXT_LOCALE` cookie so subsequent visits respect the choice.

ARIA: rendered as a `role="radiogroup"` with `role="radio"` + `aria-checked` on the two buttons. Keyboard-navigable.

## 10. Server-side rendering

`generateStaticParams()` in [`src/app/[locale]/layout.tsx`](../../design-system-preview/src/app/[locale]/layout.tsx) returns `[{ locale: "fr" }, { locale: "en" }]` so Next.js pre-renders both variants at build time. `setRequestLocale(locale)` is called early in the layout to opt the rest of the tree into static rendering.
